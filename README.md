# FORHOS

> **병원 방문 전 대기 현황을 확인하고, 증상 정보를 기반으로 더 효율적인 진료 접수를 돕는 병원 대기 관리 서비스**

---

## 프로젝트 소개

**FORHOS**는 병원에 직접 방문하기 전,

현재 대기 현황과 예상 대기 시간을 확인하고 진료 접수를 준비할 수 있는 서비스입니다.

이른 시간에 병원에 갔음에도 불구하고 긴 대기 시간을 겪었던 개인적인 경험에서 시작했습니다.

특히 지방이나 소규모 병원에서는 디지털 대기 시스템이 충분히 적용되지 않은 경우가 많다고 느꼈고, 이를 개선할 수 있는 서비스를 기획했습니다.

---

## 기획 배경

### “왜 병원에 일찍 가도 오래 기다려야 할까?”

- 이른 시간에 병원에 방문해도 긴 대기 시간이 발생
- 지방 병원에서는 대기 시스템이 잘 적용되지 않은 경우가 많음
- 현재 대기 인원이나 예상 진료 시간을 미리 알기 어려움
- 환자의 기존 질병, 복용 약 정보를 진료 전에 전달하기 어려움
- 감기, 골절 등 증상에 따라 어떤 진료과를 가야 하는지 헷갈리는 경우가 있음
- No-show로 인해 병원 운영과 다른 환자 대기 시간이 함께 영향을 받음

## 해결하고자 한 문제

| 문제 | 해결 방향 |
| --- | --- |
| 병원 방문 후 긴 대기 발생 | 방문 전 현재 대기 현황 제공 |
| 예상 진료 시간을 알기 어려움 | 평균 대기 시간 제공 |
| 접수 시 반복적인 개인정보 입력 | 사용자 정보 사전 등록 |
| 증상에 맞는 진료과 선택 어려움 | 증상 키워드 기반 진료과 분류 |
| No-show로 인한 병원 운영 문제 | No-show 발생 시 불이익 정책 구상 |

- 병원 목록 조회
- 병원별 평균 대기 시간 확인

- No-show 방지를 위한 정책 구상

## Frontend Architecture

```text
src
├─ app
│  ├─ providers
│  └─ router
├─ pages
├─ widgets
├─ features
├─ entities
└─ shared
```

| 폴더 | 역할 |
| --- | --- |
| `app` | 앱 전체 설정, 라우터, Provider 관리 |
| `pages` | 페이지 단위 화면 구성 |
| `widgets` | Header, Footer, 검색 영역 등 큰 UI 블록 |
| `features` | 로그인, 접수, 프로필 수정 등 기능 단위 UI |
| `entities` | 병원, 대기열 등 도메인 데이터와 타입 |
| `shared` | 공통 UI, 유틸 함수, route 설정 |

## 기술 선택 이유

### TypeScript

기존 JavaScript로 개발할 때는 잘못된 객체 구조나 값이 들어가도 실행 전까지 오류를 찾기 어려웠습니다.

TypeScript를 적용하면서 객체 타입을 미리 정의하고, 컴포넌트 props와 form 데이터 구조를 명확하게 관리할 수 있었습니다.

> 처음에는 타입 선언, 제네릭, 다양한 타입 문법이 어렵게 느껴졌지만
>
> 프로젝트에 조금씩 적용하면서 안전한 개발 방식에 익숙해질 수 있었습니다.

---

### TanStack React Query

병원 대기 현황, 접수 상태, 사용자 정보처럼 서버에서 받아와야 하는 데이터가 많아질 것으로 예상했습니다.

그래서 서버 상태 관리와 캐싱에 강점이 있는 TanStack React Query를 학습하고 적용했습니다.

- `useQuery`를 통한 조회 데이터 관리
- `useMutation`을 통한 등록, 수정 요청 처리
- 캐싱을 활용한 불필요한 API 요청 감소
- 서버 상태와 클라이언트 상태 분리

---

### Zod

사용자의 개인정보, 건강 정보, 복용 약 정보는 정확한 형식으로 입력되는 것이 중요하다고 생각했습니다.

Zod를 사용해 schema 기반으로 form 데이터를 검증하고, React Hook Form과 함께 사용하여 입력값 검증 로직을 일관성 있게 관리했습니다.

```tsx
zodResolver(schema)
```

---

### shadcn/ui & Tailwind CSS

서비스 전체의 UI를 정형화하고 일관성 있게 유지하기 위해 shadcn/ui를 사용했습니다.

Tailwind CSS는 컴포넌트 안에서 빠르게 스타일을 수정할 수 있어, 화면을 구현하고 다듬는 과정에서 효율적이었습니다.

- 재사용 가능한 UI 컴포넌트 구성
- Tailwind 기반 빠른 스타일링
- Radix UI 기반 접근성 확보
- 일관된 버튼, 카드, 입력 폼 디자인

## 프롬프팅 및 AI 활용

이번 프로젝트에서는 초기 구현 방향을 잡고, 미완성된 화면과 구조를 개선하기 위해 AI를 보조 도구로 활용했습니다.

단순히 코드를 생성하는 방식이 아니라, 제가 원하는 개발 기준을 명확히 전달하고 그 기준에 맞게 프로젝트를 정리하는 방식으로 사용했습니다.

### 사용한 프롬프트

```text
너가 알아서 깔끔하게 만들어주고 미완성부분 내가 쓰는 기술 스택을 기준으로.
그리고 클린 코드를 꼭 지켜주고 폴더구조는 FSD 구조로 만들어주고
너가 생각해서 비어있는 이미지는 해당 이미지에 어울리는 더미 이미지로 넣어줘.

위에 말을 기준으로 모든 부분 완성 시켜.
```

### 프롬프팅 의도

| 요청 내용 | 의도 |
| --- | --- |
| 깔끔하게 만들어줘 | UI와 코드 구조를 정돈하기 위함 |
| 미완성 부분 완성 | 기능 흐름이 끊기지 않도록 보완 |
| 내가 쓰는 기술 스택 기준 | React, TypeScript, TanStack Query 등 프로젝트 기술과 맞추기 위함 |
| 클린 코드 준수 | 유지보수하기 쉬운 코드 작성 |
| FSD 구조 적용 | `app`, `pages`, `features`, `entities`, `shared` 기준으로 역할 분리 |
| 더미 이미지 삽입 | 비어 있는 UI를 실제 서비스처럼 보이게 개선 |

### 적용한 부분

- 기존에 분산되어 있던 폴더 구조를 FSD 구조에 맞게 정리
- 페이지, 기능, 도메인, 공통 UI 역할을 분리
- 병원 서비스에 어울리는 더미 이미지와 mock 데이터를 활용
- TypeScript 타입을 기준으로 데이터 구조를 명확히 정의
- React Hook Form, Zod 기반 form 구조 설계
- Tailwind CSS와 shadcn/ui를 활용해 UI 일관성 유지
- 컴포넌트 재사용성을 고려해 공통 Button, Card, Field 등을 분리

### 느낀 점

AI를 활용하면서 중요한 것은 “무엇을 만들어줘”보다 **어떤 기준으로 만들어야 하는지 명확히 전달하는 것**이라는 점을 느꼈습니다.

처음에는 단순히 미완성 부분을 채우는 용도로 생각했지만, 프로젝트의 기술 스택, 폴더 구조, 코드 스타일, UI 방향성을 함께 제시하니 더 일관성 있는 결과를 얻을 수 있었습니다.

또한 AI가 생성한 코드를 그대로 사용하는 것이 아니라, TypeScript 타입이나 FSD 구조가 프로젝트 의도에 맞는지 직접 확인하면서 수정하는 과정이 필요하다는 점도 배웠습니다.

## 프로젝트를 통해 배운 점

이번 프로젝트를 통해 단순히 화면을 만드는 것에서 그치지 않고,

**데이터를 안전하게 다루는 방법**에 대해 많이 배울 수 있었습니다.

특히 TypeScript를 사용하면서 타입을 먼저 설계하는 습관이 생겼고, 코드 작성 중 에디터에서 오류를 바로 확인할 수 있어 개발 안정성이 높아졌습니다.

또한 TanStack React Query를 학습하면서 서버에서 받아오는 데이터와 클라이언트 내부 상태를 구분해서 관리하는 것이 중요하다는 점을 알게 되었습니다.

앞으로 실제 API 연결이 많아질수록 캐싱, invalidation, mutation 처리 방식을 더 깊게 적용해보고 싶습니다.
